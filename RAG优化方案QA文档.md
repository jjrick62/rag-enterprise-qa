## 一、Chunk 策略优化

**对应分析文档第 1 条：固定分块太粗暴**

### Q1.1
当前 FixedChunker 按 500 字符硬截断，断点可能落在句子中间、表格中间、代码块中间。你倾向哪种改进方向？
- A. 递归按标题层级切（## → ### → 段落 → 句子），确保断点始终在语义边界
- B. 引入轻量 NLP 分句器（如 spacy/nltk），先精确分句再拼到接近 chunk_size
- C. A+B 结合：先按标题切大块，超过阈值的再按句子精切
- D. 其他方案（请描述）

> 你的答案：C

> **洒家回复**：选得好。这是最务实的方案——先用标题兜底（零成本），大块再用分句器精修。需要注意的点：中文分句器别上 spacy（太重），用 `re.split(r'(?<=[。！？\n])\s*', text)` 正则就行，3 行代码解决。

### Q1.2
递归切分时，如果一个 `##` 标题下的内容本身就 3000 字，按句子切开之后，子块之间的 overlap 怎么处理？当前的 50 字符 overlap 还够用吗？

> 你的答案：要么直接量大管饱100overlap，或者

> **洒家回复**：overlap 不能拍脑袋定。应该跟 chunk_size 比例挂钩，行业惯例是 chunk_size 的 10%~20%。递归切分后子块大小不均匀——一个 `###` 标题下可能就 200 字，overlap 拉太大会把短 chunk 几乎全重叠掉。建议算法化：**overlap = min(chunk_size × 15%, 50)**。大块有足够重叠，小块不会被过度重叠。你同意吗？

### Q1.3
现在每个 chunk 的 metadata 只有 heading 字段。改成递归切分后，是否需要加字段？比如 `heading_stack: ["# 第一章", "## 1.1 小节"]` 记录完整的层级路径？

> 你的答案：加一下，适配我们做成graph rag的设想，加入语义簇的逻辑链路？可能技术上不好实现，给我说一下目前行业内方案

> **洒家回复**：目前两个主流方向——
> 
> **方向一：Chunk 链表（LlamaIndex 的方案）** 每个 chunk 存 `prev_chunk_id` 和 `next_chunk_id`，检索命中一个后自动把前后邻居也捞回来。实现简单，不需要额外 LLM 调用。
> 
> **方向二：语义簇（Graph RAG 的做法）** 用 LLM preprocess 抽实体 → 同一实体的 chunk 聚簇 → 每个簇生成 summary。检索时先匹配 summary，命中后把整簇 chunk 都拉出来。这就是你 Q3.1 选的 Graph RAG 的前置步骤。
> 
> 建议：**第一阶段先把 heading_stack 加上（改动小、零成本），第二阶段上 Graph RAG 时再补语义簇链路**。两个不冲突，heading_stack 是 Graph RAG 构建实体关系的输入之一。

## 二、检索策略优化 —— Hybrid Search + Reranker

**对应分析文档第 2 条和第 4 条：Embedding 不够精确 + BM25 互补**

### Q2.1
Hybrid Search 双路召回后，融合策略你倾向哪个？
- A. RRF (Reciprocal Rank Fusion)：两路各排 Top-N，按排名倒数加权合并
- B. 加权求和：向量相似度 × 0.7 + BM25 分 × 0.3（权重可调）
- C. 级联：BM25 先粗筛 → 向量在筛后结果里精排
- D. 其他方案

> 你的答案：a看起来像是主流的商业方案？？那我更倾向于a，或者我自己的设想相对分数截断，最高分*t=baseline，然后把扩展后的top20和baseline相减求差作为预筛选后的增强相关系数，放大差距比例，对这个进行分数求和截断.......写不下去了，这个只是在没有reranker的时候的平替，有了reranker就直接用就好了，那我们就按排名倒数对相同文档分数进行求和一股脑塞进reranker就好了

> **洒家回复**：你的直觉比 RRF 这个学术公式还务实——既然有 Reranker 做最终裁判，双路召回只管"广撒网不漏"，不管"精排序"。你的"相对分数截断"思路其实本质上是自己发明了简化版 reranker，思路是对的但对工程来说太费劲。建议最终链路简化为：
> ```
> BM25 Top-20 + 向量 Top-20 → 合并去重 → Reranker Top-5 → LLM
> ```
> 砍掉 RRF，一步到位。你之前问"为啥不直接各取 10 送 reranker"，答案就是——确实可以，而且更好。RRF 只在没有 Reranker 的时候有用。

### Q2.2
BM25 的实现方式：
- A. 自己写一个轻量 BM25（Python 纯实现，~100 行，无外部依赖）
- B. 接入 Elasticsearch / Meilisearch（重，但生产级）
- C. 用 rank-bm25 库（pip install，开箱即用）
- D. 直接用 ChromaDB 自带的全文检索（如果有的话）

> 你的答案：c，有现成的为啥不用

> **洒家回复**：同意，pip install 一行搞定。

### Q2.3
Reranker 模型选型：
- A. `BAAI/bge-reranker-v2-m3`（多语言，效果好，~1GB）
- B. `BAAI/bge-reranker-base`（轻量，~300MB，英文为主）
- C. `cross-encoder/ms-marco-MiniLM-L-6-v2`（经典英文 reranker，小）
- D. 先用 LLM 做 rerank（把 Top-10 候选全塞给 DeepSeek，让它挑最相关的 Top-3）

> 你的答案：a吧，也放进这个项目本地

> **洒家回复**：1GB 存本地（`backend/models/`），跟 BGE embedding 一个待遇。注意：这个模型是 sentence-transformers 系的 CrossEncoder，API 跟 embedder 几乎一样。

### Q2.4
Hybrid + Reranker 的完整检索链路你期望是：
```
用户问题
  → Query 改写（可选）
    → BM25 Top-20 + 向量 Top-20（双路并行）
      → RRF 融合 → 合并去重 Top-15
        → Reranker 精排 → Top-5
          → 送 LLM 生成
```
这个链路你觉得哪些步骤可以砍、哪些必须留？

> 你的答案：都留着吧

> **洒家回复**：你的态度是"先全留着，跑通了再砍"。建议把 RRF 这一步简化（见 Q2.1 的结论）——因为 reranker 在的情况下 RRF 没有任何实际作用。修正后的链路：
> ```
> 用户问题
>   → Query 改写（DeepSeek）
>     → BM25 Top-20 + 向量 Top-20（双路并行）
>       → 合并去重（直接按 doc id 去重）
>         → Reranker 精排 Top-5
>           → 送 LLM 生成
> ```
> Top-5 的"5"是按 **Reranker 的 Cross-encoder 分数**排序取的前 5 个，不是固定数字——Reranker 是阅读理解式打分，比 Embedding 的余弦相似度精确得多。

## 三、结构化/大局观优化

**对应分析文档第 3 条：LLM 只能看到 chunk，无法总揽全局**

### Q3.1
你文档里提了三种方案：Tree RAG、Graph RAG、MCP DB 服务。咱目前的 54 篇 IBM 技术文档属于哪种场景？

- B. 多实体多关系（如 API 之间的调用关系、版本兼容性），适合 Graph RAG
- C. A+B 混合
- D. 当前体量太小，不值得上这些重型方案

> 你的答案：b，炫技用，后期要上到个人主页主推项目的

> **洒家回复**：你是对的——IBM 技术文档之间确实有实体关系（API 调用链、版本依赖、平台兼容性）。定位清楚了：**Hybrid+Reranker 是第一阶段的实用优化，Graph RAG 是第二阶段的炫技优化**。后面排优先级按这个分。

### Q3.2
如果先做 Tree RAG PoC，索引结构你倾向：
- A. 直接把现有的 `#` `##` 标题层级当树节点，零额外 LLM 调用
- B. 用 LLM 为每个文档生成摘要树（多一层语义理解，但多一轮 LLM 调用）

> 你的答案：这个tree后期再考虑

> **洒家回复**：同意，聚焦第一阶段。

### Q3.3
MCP DB 方案在你文档里提了但没展开。你想象的具体场景是什么？比如——LLM 能自己写 SQL 查某张表，这个表里装什么数据？（文档元数据统计？实体关系表？Q&A 历史记录？）

> 你的答案：查文档元数据统计，有必要的话做多轮调用，这个能和graph方案混用吗

> **洒家回复**：能混用，而且是最佳实践。具体架构：
> ```
> 用户: "哪些文档提到了 watsonx.ai 且版本 >= 4.0？"
>   ↓
> LLM 识别 → 结构化查询，写 SQL:
>   SELECT title FROM doc_metadata WHERE platform='watsonx.ai' AND version >= '4.0'
>   ↓
> MCP DB 返回 3 篇文档 → 送 Graph RAG 做精确检索 → 生成回答
> ```
> MCP DB 管"筛选"，Graph RAG 管"深度理解"，分工明确。这就是你原始分析文档里第 3 条说的"让 LLM 自己写 SQL"的落地方式。

## 六、性能 & 成本

### Q6.1
Reranker 模型如果用 BGE-reranker-v2-m3 (~1GB)，你的 RTX 4060 (8GB VRAM) 能扛住——Embedding 模型 (~100MB) + Reranker (~1GB) 同时常驻内存，显存大概占 2-3GB。如果还要跑本地 LLM（如 qwen 7B），显存够呛。你的策略是？

> 你的答案：reranker和embedding做本地化，llm继续api

> **洒家回复**：最优解。4060 8G 跑 100MB Embedding + 1GB Reranker 还有 ~5G 余量，后面搞 Graph RAG 本地跑小模型也够。LLM 继续 API——DeepSeek 便宜而且是生成链路瓶颈，放云端合理。

### Q6.2
Hybrid Search 的 BM25 索引需要每次文档更新后重建。54 篇文档重建 BM25 索引大概几秒。如果后期文档量到 500+ 篇，你接受每次摄入后全量重建还是需要增量索引？

> 你的答案：有啥区别

> **洒家回复**：
> 
> | | 全量重建 | 增量索引 |
> |------|---------|---------|
> | 做法 | 删掉旧索引，从头建 | 只对新文档建索引，追加到旧索引里 |
> | 时间 | O(N) 文档数越多越慢 | O(1) 不管多少文档都差不多 |
> | 复杂度 | 一行代码 | 需要去重逻辑、版本管理 |
> | 适用 | N < 500 | N > 500 |
> 
> rank-bm25 库**不支持增量索引**——每次都得全量重建。54 篇文档重建大概 2-3 秒，500 篇也就 20-30 秒。现阶段完全够用，后期如果有增量需求再切到 Elasticsearch 这种自带增量索引的方案。

### Q7.2
Hit Rate 和 MRR 的及格线你自己定多少？
- Hit Rate@5（前 5 个检索结果中至少一个包含正确答案）：__%
- MRR（正确答案在检索结果中的平均倒数排名）：__

> **洒家回复**：
> 
> | 指标 | 行业及格线 | 行业优秀线 | 说明 |
> |------|-----------|-----------|------|

> 你目前纯向量检索跑 watsonxDocsQA 数据（30 条，每条 5 个来源全回来），粗略估计 Hit Rate@5 大概在 85%~90% 左右。加上 Hybrid + Reranker 之后目标是冲到 **Hit Rate ≥ 95%，MRR ≥ 0.8**。
