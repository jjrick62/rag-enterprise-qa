# RAG 项目文档与实现审计报告

> 审计日期：2026-06-06  
> 审计依据：当前工作区代码、测试、评估数据与 `RAG项目交接文档_2026-06-06.md`

## 一、结论

当前交接文档比 README 和 `claude_memory` 更接近真实状态，但仍包含若干未经代码验证或已经被代码反证的结论。项目现阶段不宜继续用“30/30 全通过”“唯一问题是表格检索”“heading_stack 始终为空”等表述。

优先级最高的不是继续调 RRF，而是先修复两项检索/摄入正确性问题，并让评估口径与文档一致：

1. Hybrid Search 的 BM25 分支未应用 `category_filter`，分类检索可能返回其他分类内容。
2. `/api/documents/reingest-all` 在 ChromaDB 客户端仍存活时删除数据库目录，违反交接文档记录的文件锁约束。
3. 本地 QA 文件有 30 条，但 `gen_answers.py` 只生成前 20 条；当前 RAGAS 分数也是 20 条口径。

## 二、实现问题

### P0：分类过滤在 Hybrid Search 中失效

- 位置：`backend/services/hybrid_retriever.py:65-72`
- 向量分支传入了 `category_filter`，BM25 分支只传入查询文本。
- RRF 会把未过滤的 BM25 结果与已过滤的向量结果合并。
- 影响：用户指定分类时，最终上下文仍可能出现其他分类文档，属于检索边界错误，也会污染引用和评估。
- 建议：给 `_bm25_search()` 增加分类过滤，或在 RRF 前过滤 BM25 候选。

### P0：全量重摄入端点可能破坏正在使用的 ChromaDB

- 位置：`backend/routers/documents.py:160-180`
- 路由直接 `shutil.rmtree(data/chroma_db)`，但 `pipeline._retriever` 持有活动的 ChromaDB client/collection。
- 这与交接文档“不能在 ChromaDB 连接存活时删除”的已知陷阱直接冲突。
- 删除后代码也没有重建 `ChromaRetriever`、`HybridRetriever` 或清空内存 BM25。
- 影响：重摄入可能失败、写入旧句柄、造成索引状态与 `ingestion_state.json` 不一致。
- 建议：通过 collection API 清空数据并同步清空 BM25，或关闭并完整重建 pipeline 后再摄入。

### P1：评估名义 30 条，实际仅 20 条

- 位置：`backend/gen_answers.py:40`
- `data/watsonxDocsQA_test.json` 有 30 条，代码使用 `qa_pairs[:20]`。
- `data/eval_dataset.json` 当前也是 20 条，`ragabilitytest.md` 正确写了 20 条。
- 影响：README 的“30/30”与交接文档的“生成 30 题答案”均不成立；不同报告不能直接横向比较。
- 建议：明确区分“30 条检索命中测试”和“20 条 RAGAS 生成评估”，或统一为 30 条后重跑。

### P1：`top_k` 参数被 HybridRetriever 忽略

- 位置：`backend/services/hybrid_retriever.py:58-73`
- 无论 Pipeline 请求多少候选，HybridRetriever 都固定执行双路 Top-20、RRF Top-15。
- 交接文档架构写“RRF 融合(15→20)”，后文又建议“15→30”，但当前代码始终是 15。
- 影响：调 Pipeline 的 `top_k` 不会改变候选数，容易导致错误实验结论。
- 建议：让 `search(top_k=...)` 真正控制 RRF 输出，并把双路召回数作为显式配置。

### P1：表格修复缺少测试，且打破通用 chunk_size 约束

- 位置：`backend/services/recursive_chunker.py:151-294`
- 当前 8 个 RecursiveChunker 测试全部通过，但没有任何伪表检测、转换、大表拆分或误判用例。
- 表格 chunk 允许达到 `max(chunk_size * 3, 1500)`，与现有测试和类注释中的“chunk 不超限”契约不一致。
- `_convert_pseudo_table()` 固定丢弃第 4 列以后内容，只有当前参数表适用，不能称为通用表格方案。
- 建议：补充真实 Q02 表格、无标题表格、代码块、多空格普通文本、大表、四列关键信息等测试，并在文档中明确这是“IBM 参数伪表专项转换”。

### P2：摄入不存在文件时，API 文档承诺的 404 不可靠

- 位置：`backend/routers/documents.py:63-66`
- `_file_hash(req.file_path)` 在 `try/except FileNotFoundError` 之前执行。
- 影响：不存在文件可能由 FastAPI 返回 500，而不是文档写的 404。
- 建议：先检查文件存在性，或把 hash 与 ingest 一起放入异常处理。

### P2：全量 pytest 当前失败

- `backend/ablation_test.py` 中名为 `test()` 的业务函数被 pytest 自动收集，并要求不存在的 fixture：`name`、`pipe`、`questions`。
- 结果：19 passed，1 error。
- 单独运行 `tests/test_recursive_chunker.py` 为 8 passed。
- 建议：将该函数改名为 `run_ablation_case()`，或配置 pytest 只收集 `tests/`。

## 四、对交接文档的修正

1. “heading_stack 始终为空”不成立。代码会从 Markdown 标题树生成层级，现有测试也验证了 1.2 节能保留完整路径。真正缺失的是 `char_start/char_end`，它们被统一写成 `-1`。
2. “唯一外部依赖是 DeepSeek API”只在运行时且模型/数据已经本地缓存的限定下成立。首次安装仍依赖 PyPI，缓存缺失时还涉及 HuggingFace。
3. “表修复后 RAGAS 尚未跑”正确；当前 `eval_ragas_en.txt` 的 0.610/0.770/0.550 是表修复前结果。
4. “RRF 融合(15→20)”不符合代码；当前固定输出 15。
5. “BM25 #22”与 BM25 Top-20 代码不一致。若诊断脚本看到 #22，说明它使用了不同候选规模或直接查看全量排名，文档需要注明实验口径。
6. “表格工业标准实现”表述过强。当前实现是针对空格对齐、前三列参数表的启发式转换。
