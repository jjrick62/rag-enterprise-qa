## 一、系统架构

```
用户提问 → QueryRewriter → BM25 + Vector 双路 → RRF融合(15→20) → Reranker精排(Top-5) → LLM生成(DeepSeek) → 流式返回
```

### 技术栈

| 组件 | 选型 | 位置 |
|------|------|------|
| Embedding | BAAI/bge-small-zh-v1.5 (512维) | `backend/models/` (92MB) |
| Reranker | BAAI/bge-reranker-v2-m3 (Cross-encoder) | `backend/models/` |
| 向量库 | ChromaDB 持久化 | `data/chroma_db/` |
| 关键词检索 | BM25 (rank-bm25) | `services/hybrid_retriever.py` |
| LLM 生成 | DeepSeek Chat API | 外部 |
| 评估 | RAGAS 0.4.3 (独立 venv: venv_ragas) | `services/ragas_evaluator.py` |

## 二、本会话完成的工作

### 1. 根因诊断（最重要的发现）

跑通迹脚本，逐步骤对比多个问题，发现 Faithfulness=0.566 的**真正根因不是 chunk 信噪比**：

- **纯文本题**（Q03 tokens, Q10 trust calibration）：答案在一句话里，chunk 完整，gt_overlap 86-93% → ✅ 完美
- **表格题**（Q02 tuning parameters）：6 行参数表被 `_chunk_by_sentences()` 的句子切分裂成 3 碎片，碎片 B/C 丢失表头 `Table 1: Tuning parameters`，BM25 根本无法匹配 → ❌ gt_overlap=7.7%

### 2. System Prompt 英文化（`services/prompts.py`）

中文 prompt → 英文 prompt。RAGAS 结果：

| 指标 | 中文 Prompt | 英文 Prompt | 变化 |
|------|-----------|-----------|------|
| Faithfulness | 0.566 | **0.610** | +0.044 |
| Answer Relevancy | 0.672 | **0.770** ✅ | +0.098 |
| Context Precision | 0.560 | **0.550** | -0.010 |

Answer Relevancy 已达标（≥0.70）。

### 3. 本地化

- QA 测试集 30 条 → `data/watsonxDocsQA_test.json`（不再每次从 HuggingFace 拉）
- `gen_answers.py` 优先读本地缓存
- 模型（BGE embedding + reranker）本地化 → `backend/models/`
- 唯一外部依赖：DeepSeek API

### 4. 表格 Chunk 修复（`services/recursive_chunker.py`）

参照工业标准（RAG-Architect Table Guard、KX、Unstructured.io）实现：

**新增方法：**
- `_detect_table_regions(text)` — 检测空格对齐伪表格（IBM 文档特有格式）
- `_convert_pseudo_table(lines)` — 伪表转紧凑 key-value 文本
- `_chunk_table(table_info, heading_stack)` — 表原子化，≤1500 chars 整表一个 chunk
- `_sentence_chunk(text, heading_stack)` — 原句子切分逻辑提取为独立方法

**修改方法：**
- `_chunk_by_sentences()` — 先检测表→表保护+非表句子切

### 5. 配置变更

- `chunk_size=500, overlap_ratio=0.15, max_overlap=50`
- 表容忍度：`max(chunk_size * 3, 1500)` chars

## 四、识别的剩余问题

### P0：表 chunk 进不了 Reranker Top-5

- BM25 找到但排在 #22
- Vector 完全找不到（embedding 不理解 k-v 格式）
- RRF 取并集时稀释 BM25 排名
- **建议方案**：
  1. 提高 RRF top-K（15→30），让 Reranker 多看候选
  2. RRF 给 BM25 加权（调 k 参数）
  3. 或者在 RRF 之前对 BM25 top-N 做保底注入

### P1：Context Precision 低 (0.550)

问题不只在 Q02——看 per-question 诊断数据，多个问题的 gt_overlap 低于 50%，需要逐个排查。

### P2：Answer Relevancy 达标但 Faithfulness 差 0.19

RAGAS Faithfulness 是两步法：先拆 claims 再验证。可能需要：
- 更严格的 System Prompt（减少 LLM 编造）
- 或者检查 RAGAS Judge LLM 是否过于严格

### P3：heading_stack 始终为空

`chunk()` 方法里 `char_start=-1, char_end=-1` 被注释为"递归切分后字符位置不可靠"，heading_stack 也传了空？需要排查。

## 六、已知陷阱

1. **BM25 必须从 ChromaDB 重建**：`HybridRetriever.__init__()` 里有 `_rebuild_from_chromadb()`，这是修复过的——之前这个调用缺失导致 BM25 全程空跑。
2. **Reranker sigmoid 是正确的不删**：之前有人想删 sigmoid，被阻止了。sigmoid 把 logits 映射到 [0,1]，0.68 vs 0.50 的差距正是 Reranker 工作的证明。
3. **不要搞文档级去重**：对长文档多 chunk 场景是负优化，已回滚。
4. **不要搞知识库扩展**：伪需求，已砍。
5. **venv_ragas 需要绝对路径**：RAGAS 评估必须用 `venv_ragas\Scripts\python.exe` 的绝对路径 + `SENTENCE_TRANSFORMERS_HOME` 环境变量。
6. **HuggingFace 被墙**：所有模型必须 `local_files_only=True`，`HF_HUB_OFFLINE=1`。
7. **ChromaDB SQLite 文件锁**：重摄入前必须先删 `data/chroma_db/` 目录，不能在 ChromaDB 连接存活时删除。
8. **不要用 `__file__` 在 `python -c` 中**：inline 脚本用 `os.getcwd()` 代替。
