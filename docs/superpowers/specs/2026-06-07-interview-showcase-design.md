**定位：** 面向 RAG/LLM、AI 应用与后端岗位的均衡型端到端工程项目

## 1. 目标

将当前仓库整理为一个面试官可在 30 秒内理解、在 5 分钟内验证技术深度的项目案例，同时保持现有运行链路稳定。

核心目标：

- 首屏立即说明项目价值、当前最佳指标和完整技术链路。
- 用可追溯的实验记录证明优化过程，而不是只展示最终分数。
- 同时体现检索算法、后端工程、评测治理和产品交付能力。
- 整理文档与展示资产，但不为目录美观强行迁移 `backend` 运行代码。
- 明确区分正式端到端结果、非端到端上限和受截断污染的历史实验。

## 2. 视觉方向

采用三种视觉语言的组合：

- **A 工程战报：** README 首屏和核心成果区使用深色底、青绿色高亮、工程仪表盘式信息层级。
- **C 产品案例：** 优化案例使用问题、判断、行动、结果的叙事结构。
- **B 基准实验室：** 所有 RAGAS 数据图使用白底、严谨坐标、明确图例和实验口径注释。

视觉约束：

- 不使用通用紫色 AI 渐变、机器人图片或装饰性假数据。
- 图表必须能在 GitHub 浅色和深色界面中清晰阅读。
- SVG 为主格式，PNG 用于预览或不支持 SVG 的场景。
- 数据图只展示仓库内可追溯的真实实验结果。

## 7. 代表性案例结构

每个案例采用统一模板：

1. **现象：** 指标、问题样本或日志表现。
2. **错误假设：** 最初可能误判的方向。
3. **证据：** 对照实验、完整上下文或代码路径。
4. **根因：** 检索、生成、Judge 或评测传输中的具体缺陷。
5. **修复：** 最小范围改动。
6. **结果：** 指标变化、行为验证和剩余风险。

重点案例：

- RAGAS 审查截断漏洞：生成模型使用完整 chunk，但 Judge 只收到 200 字前端摘要。
- 自适应过滤阈值：0.80 提高 Faithfulness，却降低 Answer Relevancy 与 Context Precision；0.75 综合最佳。
- Prompt 过度拒答：上下文存在完整调参表，但模型因约束过严回答“无信息”。
- Q00/Q02 拆因：正确答案被 Judge 误判与生成模型真实拒答不能混为一个问题。

## 8. 验证与验收

完成后必须满足：

- README 在 GitHub 桌面宽度与窄屏下可读。
- 所有本地 Markdown 链接和图片路径有效。
- SVG 中的数值与权威数据源一致。
- 架构图中的组件、阈值和调用顺序与当前代码一致。
- 文档移动后没有失效引用。
- 现有测试保持通过，启动命令不变。
- 根目录只保留面试官应优先看到的入口文件。
- 临时设计目录 `.superpowers/` 不进入 Git。
- 变更分阶段提交，最后推送到 `origin/main`。

## 9. 非目标

- 不在本次展示重构中切换 Embedding 模型。
- 不重新运行高成本 RAGAS 全量评测，除非发现现有报告不足以支持图表。
- 不将项目强迁移到 `src/` 布局。
- 不改写检索算法、生成 Prompt 或线上 API 行为。
- 不删除原始评测报告和历史交接材料。
